# 📘 FabricLab MVP – Terminal & Lab Engine Documentation

## 1. Overview

This document captures the evolution and current design of the **FabricLab MVP terminal + lab evaluation system**.

The goal of the system is to simulate **real network troubleshooting workflows**, not rigid command execution.

---

## 2. Core Architecture

### 2.1 Components

* **Terminal (xterm.js)**

  * Handles user input
  * Echoes commands
  * Dispatches commands to handler logic

* **State Store (Zustand)**

  * Holds lab state
  * Tracks configuration and user behavior

* **Lab UI (Left Panel)**

  * Displays instructions
  * Displays status (in-progress / warning / success)

---

## 3. State Model

Current Zustand store structure:

```ts
{
  pfcEnabled: boolean,     // Current system state
  hasVerified: boolean,    // Whether user verified after change
  mistakeCount: number     // Tracks incorrect commands
}
```

---

## 4. Command Handling Logic

### 4.1 Supported Commands

#### 1. show dcb pfc

* Displays current PFC state
* If `pfcEnabled === false` → sets `hasVerified = true`

#### 2. disable pfc

* Sets `pfcEnabled = false`

#### 3. Unknown commands

* Increment `mistakeCount`

---

## 5. Evolution of Logic (Important)

### Phase 1 (Initial)

* Command echo only
* No real state

---

### Phase 2 (State introduction)

* Introduced:

  * `pfcEnabled`
* Commands started modifying state

---

### Phase 3 (Verification logic)

* Added:

  * `hasVerified`
* Lab required:

  ```
  disable pfc → show dcb pfc
  ```

⚠️ Problem:

* Order too strict
* Unrealistic behavior

---

### Phase 4 (Strict sequencing attempt)

* Introduced sequence enforcement

⚠️ Problems:

* Broke natural troubleshooting
* Penalized valid user exploration

---

### Phase 5 (Final – Current Model ✅)

Switched to:

## 👉 State-based + Behavior-based evaluation

---

## 6. Final Lab Completion Logic

### Completion Condition:

```ts
pfcEnabled === false AND hasVerified === true
```

---

## 7. Behavior Tracking

### mistakeCount

Incremented when:

* Unknown command
* Invalid syntax

---

## 8. Lab Result Classification

### 🟢 SUCCESS (Clean Execution)

```
mistakeCount === 0
```

### 🟡 WARNING (Completed with mistakes)

```
mistakeCount > 0
```

## 11. Design Principles (Critical)

### 1. Do NOT enforce strict command order

Real engineers:

* Check state first
* Try commands
* Make mistakes

---

### 2. Reward correct final state

Lab completion is based on:

* Configuration correctness
* Verification step

---

### 3. Track behavior quality

Not just outcome, but:

* How user got there

---

### 4. Keep logic minimal (MVP rule)

Avoid:

* Over-engineering
* Complex state machines

---

## 12. Known Edge Cases (Handled)

| Scenario                 | Behavior |
| ------------------------ | -------- |
| Verify before disable    | Ignored  |
| Multiple verify commands | Allowed  |
| Mistakes before success  | Counted  |
| Mistakes after success   | Ignored  |
