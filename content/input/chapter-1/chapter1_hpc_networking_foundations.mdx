              const isCorrect = oi === q.correct
              let bg = "#FFFFFF", border = "#E5E7EB", color = "#374151"
              if (isSubmitted) {
                if (isCorrect) { bg = "#F0FDF4"; border = "#22C55E"; color = "#166534" }
                else if (isSelected && !isCorrect) { bg = "#FEF2F2"; border = "#EF4444"; color = "#7F1D1D" }
              } else if (isSelected) {
                bg = "#EFF6FF"; border = "#3B82F6"; color = "#1D4ED8"
              }
              return (
                <div
                  key={oi}
                  onClick={() => !isSubmitted && setAnswers({ ...answers, [q.id]: oi })}
                  style={{
                    border: `0.5px solid ${border}`, borderRadius: 8,
                    padding: "10px 14px", marginBottom: 8, cursor: isSubmitted ? "default" : "pointer",
                    background: bg, color, fontSize: 13, lineHeight: 1.5,
                    transition: "all 0.15s",
                  }}
                >
                  <span style={{ fontWeight: 600, marginRight: 8 }}>{["A", "B", "C"][oi]}.</span>
                  {opt}
                </div>
              )
            })}
            {!submitted[q.id] && answers[q.id] !== undefined && (
              <button
                onClick={() => setSubmitted({ ...submitted, [q.id]: true })}
                style={{
                  padding: "8px 18px", fontSize: 13, background: "#1E293B",
                  color: "#FFFFFF", border: "none", borderRadius: 7, cursor: "pointer", marginTop: 4,
                }}
              >Submit answer</button>
            )}
            {submitted[q.id] && (
              <div style={{
                marginTop: 10, padding: "10px 14px",
                background: answers[q.id] === q.correct ? "#F0FDF4" : "#FEF2F2",
                borderRadius: 8, fontSize: 13, lineHeight: 1.65,
                color: answers[q.id] === q.correct ? "#166534" : "#7F1D1D",
              }}>
                <strong>{answers[q.id] === q.correct ? "Correct." : "Incorrect."}</strong> {q.explanation}
              </div>
            )}
          </div>
        </div>
      ))}
      {Object.keys(submitted).length === questions.length && (
        <div style={{
          background: score === questions.length ? "#F0FDF4" : score >= 3 ? "#FFFBEB" : "#FEF2F2",
          border: `1px solid ${score === questions.length ? "#22C55E" : score >= 3 ? "#F59E0B" : "#EF4444"}`,
          borderRadius: 12, padding: "16px 20px", textAlign: "center",
        }}>
          <div style={{ fontSize: 28, fontWeight: 700, color: "#111827", marginBottom: 4 }}>{score}/{questions.length}</div>
          <div style={{ fontSize: 14, color: "#374151" }}>
            {score === questions.length
              ? "Perfect score. You're ready for Chapter 2: InfiniBand Architecture."
              : score >= 3
              ? "Good understanding. Review the questions you missed before continuing."
              : "Re-read the relevant sections above before moving to Chapter 2."}
          </div>
        </div>
      )}
    </div>
  )
}

## 1. Why HPC networking is unlike anything you've done before

You've spent years mastering TCP/IP, BGP, spanning tree, QoS, and MPLS. You can troubleshoot a flapping OSPF adjacency in your sleep. But when you walk into an AI data center for the first time, something immediately feels wrong — none of the familiar mental models apply cleanly.

In enterprise networking, a 1 ms spike in latency is a footnote in a monitoring dashboard. In an AI training cluster running a large language model across 1,024 GPUs, that same 1 ms spike causes every GPU in the job to sit idle waiting for one straggler — and that idle time compounds across hundreds of training steps per hour. **The fabric isn't just moving data; it is the synchronisation mechanism for a massively parallel computation.**

## 2. The AI training problem that drives everything

AI model training is a mathematical process called gradient descent. The model makes predictions, measures how wrong it was (the "loss"), then adjusts hundreds of billions of internal parameters in the direction that reduces that loss. Repeat millions of times.

The problem is scale. A 70B parameter model cannot fit in a single GPU's memory — it must be split across dozens or hundreds of GPUs simultaneously. After each training step, all GPUs must agree on the updated parameters. This requires **AllReduce**: every GPU exchanges gradient updates with all other GPUs, they are summed, and the result goes back to every GPU before the next step can begin.

**That synchronisation point is where the network becomes the bottleneck.** Step through the sequence below to see exactly why:

<AllReduceBarrier />

The numbers that matter for the network design:

| Metric | Enterprise standard | HPC requirement | Why |
|--------|---------------------|-----------------|-----|
| Latency | 1–5 ms | <2 µs | AllReduce stall × 1000s of steps/day |
| Packet loss | <0.1% acceptable | Zero tolerance | One retransmission stalls the entire job |
| Bandwidth per node | 10–100 Gbps | 3,200 Gbps (DGX H100) | Full bisection needed for AllReduce at scale |
| Congestion response | QoS queuing | ECN + PFC lossless | Dropping packets is catastrophic |

## 4. The NIC is not just a NIC — understanding HCAs and SuperNICs

In enterprise networking, a NIC is a commodity component — it moves packets between the server's kernel and the wire. In HPC, the network adapter is an **active compute element** with its own processor, memory, and instruction set. This distinction fundamentally changes how you think about the fabric.

<NICExplorer />

### The RDMA data path — what actually happens when a GPU sends data

This is the most important pipeline to understand in this entire course. Follow it step by step:

<CalloutBox type="success" title="The key insight">
The CPU is only involved to post the Work Request and read the Completion — typically a few hundred nanoseconds of CPU time. The actual data movement (potentially gigabytes) happens entirely between the HCA and GPU memory with zero CPU involvement. This is how 400 Gb/s with sub-2 µs latency is achievable.

## 5. The three networks inside every DGX deployment

A DGX BasePOD or SuperPOD is not one network — it is three physically separate networks running simultaneously, each with completely different requirements.

<ThreeNetworkDiagram />

### Network 1 — Compute fabric

**Purpose:** GPU-to-GPU training traffic. AllReduce collectives, tensor transfers, RDMA operations.

**Requirements:** Lossless (zero packet drops), ultra-low latency (sub-2 µs), full bisection bandwidth (every GPU simultaneously at line rate), non-blocking (no oversubscription).

**Hardware (DGX H100 reference):** 8 × ConnectX-7 per node (400G each = 3.2 Tb/s per node). Leaf layer: QM9700 Quantum-2 NDR. Spine layer (larger clusters): QM9700 or Q3400 Quantum-X800.

### Network 2 — Storage fabric

**Purpose:** High-performance parallel storage access — training datasets, checkpoints, model artifacts.

**Requirements:** High bandwidth (200–400 GbE), low latency for random I/O, RoCEv2 or NFS-over-RDMA for storage protocols. **GPUDirect Storage (GDS)** allows the GPU to read training data directly from NVMe storage without CPU or system RAM involvement.

**Hardware:** Dedicated Ethernet NIC (or BlueField-3 DPU handles storage). Switches: SN4600C or SN5600. Storage: BeeGFS, WEKA, VAST Data, DDN.

### Network 3 — Management OOB

**Purpose:** Cluster management, health monitoring, provisioning, firmware updates, BMC/IPMI access, UFM/BCM orchestration.

**Hardware:** Standard enterprise 1 GbE / 10 GbE switches. Completely separate physical switches from compute and storage.

<CalloutBox type="danger" title="Critical rule — never cross-connect these networks">
Routing compute RDMA traffic through the management switch — even temporarily during initial bring-up — can cause PFC deadlocks or congestion events that are extremely difficult to diagnose. The three networks must be physically separated from day one, every time.
</CalloutBox>

## 6. Inside the DGX node — the full internal architecture

Understanding what is inside a DGX node explains why the external fabric is designed the way it is. The node itself is already a high-performance interconnect problem — 8 GPUs must communicate with each other at terabytes-per-second before a single bit leaves the chassis.

<DGXInternalDiagram />

### The NVSwitch layer — intra-node GPU fabric

| Generation | DGX system | BW per GPU | Total node BW |
|-----------|-----------|------------|---------------|
| NVSwitch gen2 | DGX A100 | 600 GB/s | 4.8 TB/s |
| NVSwitch gen3 | DGX H100/H200 | 900 GB/s | 7.2 TB/s |
| NVSwitch gen4 | DGX B200 / GB200 NVL72 | 1,800 GB/s | 14.4 TB/s |

NVSwitch is NVIDIA's proprietary chip creating a full non-blocking all-to-all fabric between all 8 GPUs. GPU 0 can send to GPU 7 at 900 GB/s simultaneously with every other pair — non-blocking. This is why **intra-node** communication (tensor parallelism within a node) is 18× faster than **inter-node** communication over InfiniBand.

### Why rail-optimised topology matters

The 1:1 GPU-to-NIC ratio and rail design provides two critical benefits:

**Maximum AllReduce performance:** When 32 DGX H100 nodes run AllReduce, GPU 0 on every node communicates via Rail-0 switches, GPU 1 via Rail-1, and so on. Each rail carries exactly 1/8 of the total AllReduce traffic with no switch fan-in conflict.

**Fault isolation:** If Rail-3's leaf switch fails, only GPU 3 on every node loses inter-node connectivity — 7 GPUs per node remain fully operational. If all 8 NICs shared a single uplink switch, the entire node would go dark.

## Chapter summary

The foundational mental models to carry into every remaining chapter:

1. **HPC networking exists to serve a synchronisation barrier.** AllReduce is a collective operation — the fabric's job is to make that barrier as fast as possible, because every GPU waits for the slowest link.

2. **The DGX node is already an HPC fabric problem.** NVSwitch solves intra-node at 900 GB/s; ConnectX-7 and InfiniBand solve inter-node at 400G per GPU. The 18:1 bandwidth ratio directly drives the 3D parallelism strategy.

3. **The NIC (HCA) is an active compute element.** It bypasses the kernel, DMA's directly to and from GPU memory, and processes RDMA verbs in hardware. Think of it as a co-processor, not a commodity component.

4. **Three networks, always physically separate.** Compute (InfiniBand/RoCEv2), storage (Ethernet + GDS), management (OOB). Never mix them.

5. **Rail-optimised topology is the standard.** 1 GPU : 1 NIC : 1 leaf switch rail. Maximum bandwidth, maximum fault isolation.

6. **Cable choice is a design decision.** DAC for in-rack, AOC for inter-rack, pluggable fiber for long reach. Generation and connector compatibility must be verified before ordering.
