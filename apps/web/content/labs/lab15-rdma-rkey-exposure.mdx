## Scenario

A security audit has flagged a critical finding on a multi-tenant GPU cluster: a
TenantB engineer was able to read memory from a TenantA GPU node using a guessed
RKEY value. The cluster runs DGX H100 nodes with ConnectX-7 NICs. The storage
network is RoCEv2 over Ethernet — no InfiniBand.

You have been handed a test environment that reproduces the vulnerable configuration:

- `tenanta-node`: A TenantA compute node that has pre-allocated a Memory Region
  and is listening for legitimate RDMA connections from TenantA peers.
- `tenantb-node`: A TenantB compute node. It has been pre-loaded with `rkey_scan`,
  a tool that iterates RKEY values and attempts RDMA reads.
- `leaf-switch`: The SN5600 Spectrum-X leaf switch both nodes connect to.
- `ufm-server`: The UFM Enterprise server managing the fabric.

Your task: reproduce the attack, identify both misconfigurations, fix them, and
confirm the attack fails after hardening.

## Task 3 — Identify the Two Misconfigurations

You have now seen the attack succeed. Your job is to identify exactly why it worked.
There are two independent misconfigurations — either one alone would be sufficient
to prevent the attack, but both should be fixed.

<LabTerminal nodeId="tenanta-node" prompt="tenanta-node:~$">
```
# Check GID filtering status on the ConnectX-7 NIC:
show gid filter

# This shows the current value of ROCE_ADDR_FILTER_ENABLE
# and whether tenantb's GID is in the allowed set
```
</LabTerminal>

<LabHint level={1}>
Two misconfigurations enabled this attack: (1) GID filtering is disabled — Tenant B's
NIC was allowed to establish a QP connection to Tenant A's NIC with no restriction.
(2) The RKEY is low-entropy — once the QP connection was established, the attacker
only needed to try a small range of values. Fix both.
</LabHint>

<LabHint level={2}>
`show gid filter` will show `ROCE_ADDR_FILTER_ENABLE: 0 (DISABLED)`. This means any
remote GID can connect. The factory default for ConnectX-7 is disabled. This is the
first fix: `enable gid filter`. The second fix is `ibv_reg_mr rotate` to force a new
high-entropy RKEY.
</LabHint>

<LabCheck id="check-misconfiguration-identified" title="Both misconfigurations identified">
Pass: `show gid filter` run on `tenanta-node` shows GID filtering is DISABLED.
Condition `gidFilteringIdentified` is set.
</LabCheck>

<LabHint level={1}>

</LabHint>

<LabCheck id="check-gid-filter-enabled" title="GID filtering enabled">
Pass: `show gid filter` after the fix shows `ROCE_ADDR_FILTER_ENABLE: 1 (ENABLED)`

## Task 5 — Rotate the RKEY

Apply the second fix: force a new RKEY by deregistering and re-registering the
Memory Region. This invalidates the old (guessed) RKEY and assigns a new one from
a high-entropy pool.

<LabTerminal nodeId="tenanta-node" prompt="tenanta-node:~$">
```
# Rotate the Memory Region RKEY:
ibv_reg_mr rotate

# Verify the new RKEY has high entropy:
show mr info after
```
</LabTerminal>

<LabHint level={1}>
After rotation, `show mr info after` should show a new RKEY value in the range
`0xA0000000–0xFFFFFFFF` — a high-entropy value that would take on average 2 billion
guesses to find, vs the original which needed only 39. Note that any legitimate
TenantA peer that was using the old RKEY will need the new value via the application's
connection setup protocol.
</LabHint>

<LabCheck id="check-rkey-rotated" title="RKEY rotated to high-entropy value">
Pass: `show mr info after` shows a new RKEY value >= `0xA0000000`. Condition
`rkeyRotated` is set.
</LabCheck>

## Debrief

### The two misconfigurations

**(a) GID filtering disabled (ROCE_ADDR_FILTER_ENABLE=0)**
The factory default for ConnectX-7 is to accept QP connections from any remote GID.
This is appropriate in a single-tenant environment where all nodes are trusted. In a
multi-tenant cluster, it means any node can connect to any other node's RDMA endpoint
and attempt RDMA operations — subject only to RKEY validation.

The fix — `ROCE_ADDR_FILTER_ENABLE=1` — should be applied at provisioning time and
treated as a mandatory hardening step for every multi-tenant RoCEv2 deployment. It
costs nothing in performance. The check is done at QP establishment, not per-packet.

**(b) Low-entropy RKEY (predictable kernel seed)**
With GID filtering disabled, the RKEY was the only barrier. A 32-bit RKEY should
require 2 billion guesses on average. The pre-allocated MR had RKEY `0x00000027` —
only 39 guesses. This is a real vulnerability present in several Linux kernel versions
where the RKEY counter is initialised from a low-entropy source.

Check your own systems: run `show mr info` equivalent (`ibv_reg_mr` test tool) on
a freshly booted node and look at the RKEY values assigned to the first few MRs.
If they are small numbers (< 0x0000FFFF), your kernel has this issue. Ensure
`/proc/sys/kernel/randomize_va_space` is set to 2.

### Why either fix alone is insufficient

If only GID filtering is applied: RKEY rotation is not strictly necessary because a
blocked QP means the attacker cannot present any RKEY at all. But if GID filtering
is ever accidentally disabled (firmware upgrade, misconfiguration), the low-entropy
RKEY is immediately exploitable again. Both fixes in combination provide defence in
depth.

If only RKEY rotation is applied: a high-entropy RKEY makes scanning impractical but
not impossible given enough time or a side-channel. GID filtering eliminates the
attack surface entirely — no QP, no opportunity to scan.

### The enforcement hierarchy

```
Layer 1 (fabric): PKey partitioning  — blocks cross-tenant QPs at HCA hardware
Layer 2 (NIC):    GID filtering      — blocks unrecognised remote GIDs at NIC ASIC
Layer 3 (app):    RKEY entropy       — makes guessing impractical if layers 1+2 fail
```

All three layers must be deployed. In RoCEv2 (Ethernet) environments, Layer 1 (PKey)
is not available — GID filtering (Layer 2) is your primary control. In InfiniBand
environments, PKeys provide Layer 1 and GID filtering is an additional defence.
